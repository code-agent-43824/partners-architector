# Демо для Дмитрия Грица и архитекторов — сценарий (10–12 минут)

Цель демо: получить честную обратную связь «туда ли мы идём», показывая
продукт, который уже снимает рутину и собирает партнёрское соглашение —
не выглядя прототипом. Тезис: «мы не заменяем архитектора — мы убираем
из сессии всё, что не разговор».

## Подготовка (накануне; ~15 минут)

1. Пересоздать демо-кейс на сервере (эталонное состояние):

   ```bash
   podman exec -e DEMO_ARCHITECT_PASSWORD='<серверный пароль>' psa-api \
     pnpm --filter @psa/api db:seed:demo
   ```

   Логин демо-аккаунта: `demo@partners-architector.local` (пароль — серверный,
   не в репозитории). Кейс: **«Кофейня „Третье место“ (демо)»** — 3 партнёра,
   первичная сессия в работе, 24/30 согласовано, «тяжёлых не закрыто: 3».
   Повторный запуск команды в любой момент сбрасывает кейс к эталону
   (правки, сделанные на прошлой репетиции, исчезают).

2. Открыть вкладки и проверить:
   - [ ] вход под демо-аккаунтом работает; в списке — только демо-партнёрство;
   - [ ] сценарий сессии открывается, оглавление показывает статусы;
   - [ ] «Собрать соглашение» открывается; «Скачать PDF» и «Скачать DOCX»
         отдают файлы (первый PDF после рестарта api занимает ~2 с — прогреть);
   - [ ] DOCX открывается в Word/LibreOffice на машине, с которой идёт показ.
3. Закрыть лишние вкладки/уведомления; масштаб браузера 100–110%.

## Сценарий (тайминг ориентировочный)

**0:00–1:00. Контекст.** Одна фраза о продукте: «Помощник партнёрской
сессии по вашей методике: 30 вопросов, фиксация договорённостей, сборка
соглашения. Всё развёрнуто на нашем сервере, никаких внешних облаков».

**1:00–2:30. Кейс (+ режим подсказок).** После входа появится экран
«Режим подсказок» — прочитайте его вслух: он сам объясняет экран и
называет будущие функции (это и есть онбординг продукта; подсказки
появляются по одному разу на каждый новый экран и отключаются галочкой
в меню пользователя). Затем: список партнёрств → «Кофейня „Третье
место“ (демо)»: 3 партнёра с ролями, первичная сессия в работе.
Акцент: партнёрство — живой кейс, а не пустой экран. Подсказка перед
карточкой партнёрства сама расскажет про будущие кабинеты партнёров и
предсессионный опросник — не нужно держать это в голове.

**2:30–5:00. День сессии (ядро методики).** Открыть сценарий:
- оглавление по контурам, точки-статусы, метки «тяжёлый», счётчики
  («Согласовано: 24/30 · Тяжёлых не закрыто: 3»);
- фокус-режим: один блок на экране, ←/→ с клавиатуры;
- вживую дозаполнить блок **№24** («инструкция по работе с собой»):
  допечатать пару слов (автосохранение — «Сохранено»), поставить
  «согласовано», отметить подтверждения всех троих → бейдж
  «✓ подтверждено всеми», счётчик наверху обновился.

**5:00–6:00. Предохранитель методики.** На любом блоке выбрать статус
«неактуально» → модальное окно с предупреждением методики и обязательной
причиной → **отменить**. Показать блок №18, уже помеченный неактуальным
с причиной. Тезис: «система не даёт молча сбежать от трудного вопроса».

**6:00–7:00. История версий.** Блок №2 → «История версий»: две версии с
пометками, «Откатить» (текущая формулировка сохранится) → откатить и
вернуть. Тезис: «зафиксированное не теряется» (плюс ночные бэкапы).

**7:00–8:00. Доли.** Блок №5: ручные доли 50/30/20, живая проверка
«Итого: 100%», переключатель «Рассчитать доли (Калькулятор Грица) —
скоро»: «калькулятор — следующий этап, посчитаем вклад трёх капиталов по
вашей методике». Блок №6: «смысл долей» — прибыль/владение/убытки, но не
голосование.

**8:00–10:00. Главное — соглашение.** «Собрать соглашение»:
- титул (участники, дата, готовность), 4 «Принципа», 30 разделов в
  порядке методики, «неактуально» с причиной, таблица долей в §5;
- «Скачать PDF» — открыть файл;
- «Скачать DOCX» — открыть в Word, показать что форматирование живое и
  документ можно дорабатывать. Тезис: «то, что раньше собиралось руками
  за вечер, — одна кнопка».

**10:00–11:00. Что дальше (без обещаний сроков).** Калькулятор Грица
(внутри блока №5), матрица полномочий (блок №15), чек-лист легализации,
кабинет клиента с предсессионным опросником, AI-помощник на собственном
сервере заказчика (без внешних облаков).

**11:00–12:00. Вопросы к Дмитрию.**
1. Формулировки 30 блоков сейчас выверены по рабочей тетради — где мы
   промахнулись по смыслу или тону?
2. Что в дне сессии самое болезненное из того, чего вы ещё не увидели?
3. Каким должен быть калькулятор, чтобы вы ему доверяли?

## Запасные ходы

- Нет интернета у докладчика → демо целиком работает с одного ноутбука по
  открытым вкладкам; PDF/DOCX скачаны заранее как резерв.
- Вопрос «а если партнёров больше трёх?» → лимит 5, тетрадь рассчитана
  до 3 (шаблоны на 4–5) — показываем на 3.
- Вопрос про безопасность → self-hosted, изоляция по архитектору, бэкапы,
  санитизация текста, файлы экспорта не хранятся на сервере.

## После демо

- Зафиксировать обратную связь в HANDOFF (решения владельца — в
  «Owner decisions»); скорректировать приоритет Фазы 2/3 по реакции.
- При необходимости — `db:seed:demo` ещё раз, чтобы вернуть эталон.
